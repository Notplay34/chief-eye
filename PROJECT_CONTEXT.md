# PROJECT_CONTEXT

Главный документ бизнес-логики проекта. Если код и старые заметки расходятся, ориентироваться нужно на этот файл и на текущую реализацию.

## 1. Назначение системы

Система обслуживает два офлайн-павильона возле МРЭО и сводит их работу в один контур:

- оформление документов;
- изготовление номеров;
- учёт оплат и кассовых операций;
- учёт сотрудников и ролей;
- прозрачная операционная аналитика для владельца.

Цель не в “универсальной CRM”, а в простой и устойчивой внутренней системе для ежедневной работы.

## 2. Бизнес-контур

### Павильон 1

Оператор:

- заполняет форму клиента и автомобиля;
- выбирает документы и услуги;
- указывает госпошлину;
- отмечает необходимость изготовления номера;
- принимает оплату;
- печатает документы.

Результат работы павильона 1 — единый заказ с деньгами, документами и дальнейшим маршрутом.

### Павильон 2

Оператор работает через web-интерфейс и видит:

- список заказов с номерами;
- текущий статус изготовления;
- доплаты за номера;
- склад заготовок;
- кассу номеров;
- готовность заказа к выдаче.

Все действия павильона 2 меняют состояние того же заказа.

## 3. Финансы

Система разделяет:

- госпошлину;
- доход павильона 1;
- доход павильона 2.

Фиксируются:

- оплаты по заказу;
- доплаты за номера;
- строки кассы павильона 1;
- строки кассы павильона 2;
- выплаты оператору номеров;
- открытие и закрытие смен.

Госпошлина учитывается отдельно и не считается доходом бизнеса.

## 4. Аналитика

У директора есть web-аналитика по основным управленческим показателям:

- доход;
- оборот;
- госпошлина;
- количество заказов;
- средний чек;
- динамика по месяцам;
- квартальный срез;
- результат по сотрудникам;
- статусы заказов;
- топ услуг.

Аналитика строится из данных заказов и платежей, а не из отдельной витрины.

## 5. Архитектура

### Backend

- `FastAPI`
- `SQLAlchemy`
- `PostgreSQL`

### Frontend

Статические страницы и JS-модули для рабочих ролей:

- форма документов;
- касса и смены;
- номера;
- склад;
- аналитика;
- аккаунты и админка.

### Инфраструктура

- `nginx`
- `systemd`
- `deploy/check_stack.sh` для smoke-проверки после деплоя

## 6. Основные сущности

| Сущность | Назначение |
|---------|------------|
| `orders` | единый заказ |
| `payments` | платежи по заказам |
| `employees` | сотрудники и роли |
| `cash_shifts` | смены по павильонам |
| `cash_rows` | касса павильона 1 |
| `plate_cash_rows` | касса павильона 2 |
| `plate_stock` | остаток заготовок |
| `plate_reservations` | резервы под заказы |
| `plate_defects` | брак |
| `plate_payouts` | выплаты оператору номеров |
| `form_history` | история заполнения формы |

## 7. Жизненный цикл заказа

Текущие статусы:

- `CREATED`
- `AWAITING_PAYMENT`
- `PAID`
- `PLATE_IN_PROGRESS`
- `PLATE_READY`
- `COMPLETED`
- `PROBLEM`

Заказ проходит только допустимые переходы. Проверка переходов живёт в backend-сервисах.

## 8. Роли

### `ROLE_ADMIN`

- полный доступ;
- аналитика;
- админка;
- управление аккаунтами;
- оба павильона.

### `ROLE_MANAGER`

- оба павильона;
- форма;
- касса павильона 1;
- номера;
- касса номеров;
- склад;
- просмотр и рабочие операции по заказам;
- без аналитики и без управления аккаунтами.

### `ROLE_OPERATOR`

- форма документов;
- создание заказа;
- приём оплаты;
- печать документов;
- касса павильона 1.

### `ROLE_PLATE_OPERATOR`

- список заказов с номерами;
- смена статусов изготовления;
- доплата за номера;
- касса номеров;
- склад.

## 9. Принципы

- один заказ — центральная сущность;
- PostgreSQL — единственный источник правды;
- интерфейсы не хранят бизнес-состояние локально;
- роли и доступы определяются на backend;
- документация должна описывать только то, что реально есть в коде.

## 10. Поля формы

Единая форма заказа покрывает:

- клиента или юрлицо;
- продавца;
- доверенное лицо;
- данные автомобиля;
- набор документов;
- госпошлину;
- количество номеров;
- суммы по документам и ДКП.

Шаблоны документов хранятся в `templates/` в корне проекта.
